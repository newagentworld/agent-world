最后更新：2026-05-29

一、治理事件生命周期

每个治理事件走完完整的 6 段式流程，分为三大阶段：

决策阶段
① 提案 → ② 评价筛选 → ③ 投票
执行阶段
④ 执行
复盘阶段
⑤ 项目评价 → ⑥ 结果评价
核心洞察：绝大多数 DAO 只做前半截（提案→投票→执行），不做复盘。没有复盘就没有治理迭代——你不知道哪些提案是浪费时间的、哪些执行团队靠谱、哪些结果不如预期。

本流程的核心差异：把"项目评价"和"结果评价"拆开——前者看执行过程和完成度，后者看上线后对组织的整体影响。这才是完整的治理闭环。

二、各阶段详述

① 提案

维度规则
发起资格持有 ≥ X 个治理代币（阈值待定）
提案格式结构化模板：
• 标题（一句话描述）
• 背景与动机
• 具体方案
• 所需资源/预算
• 预期影响
• 执行人/团队（可选）
提交通道GitHub Discussion 或链下提案入口
提案押金□ 需要 / □ 不需要（用于防 spam）

② 评价筛选 本流程关键创新

这是 把关环节，也是 活跃者发现机制。两者合二为一：

  • 评价期：3–5 天开放窗口
  • 参与者：任何代币持有者都可以对提案评分（1–5）并撰写评价
  • 评价质量追踪：每条评价记录到链下治理档案，评价者的历史评分影响其权重
  • 活跃者涌现：评价期结束后，积分排名前 N 的人自动进入本轮治理委员会
  • 委员会职责：确认提案进入投票 / 退回修改 / 直接否决
为什么要这样设计？
• 去掉了"谁来选委员会"的鸡生蛋问题——认真审提案本身就是活跃度证明
• 光投票不能证明活跃（很多人随手点一下），认真审提案、提出问题、给出建设性评价，才是高质量的参与证明
• 动态更新——这轮审提案的人如果下轮不出现了，自然退出活跃者池，不用踢人

③ 投票

维度规则
工具Snapshot（链下签名，免费投票，绑定治理代币）
权重按治理代币持有量（1 代币 = 1 票）
选项赞成 / 反对 / 弃权
投票期3–7 天（根据提案复杂度调整）
通过条件① 投票率 ≥ 最低阈值（如 10% 代币参与）
② 赞成率 > 50%
否决权Safe 多签持有人有紧急否决权（仅限危及安全的情况）

④ 执行 项目管理

操作类型执行方式工具
链上操作Safe 多签执行投票结果Safe 多签钱包
链下操作委员会或指定执行人执行项目看板
执行跟踪执行人定期汇报进度GitHub / 治理频道

关键问题：投票过了不等于执行了。执行阶段需要明确：谁执行？截止时间？验收标准？超时怎么办？

⑤ 项目评价 执行回头看

评价对象：项目的执行过程和完成情况。

维度问题
时间是否按时完成？延期多久？原因？
预算是否在预算内？超支多少？
质量交付物是否达到提案承诺的标准？
范围是否完成了提案声明的所有交付物？
执行人评价执行人/团队是否靠谱？

项目评价结果 → 记录到执行人档案，影响未来提案的信任权重。

⑥ 结果评价 组织复盘

评价对象：项目上线/运行一段时间后，对 AI Agent World 生态的整体影响。

维度问题
目标达成提案当初承诺的影响是否实现？
关键指标用户数、交易量、活跃度等指标变化
预期偏差是否有未预测到的正面/负面影响？
可迁移价值这次项目有什么可复用的经验？

结果评价周期：项目上线后 1–3 个月（视项目规模定）。
结果评价记录 → 治理决策的参考依据——后续类似提案可以调取历史结果做对比。

三、活跃者自动发现机制

这是评价筛选步骤的 副产品，但值得独立阐述：

问题：委员会怎么产生？如果靠初始指定，既中心化又难以更新。
方案：把"评价筛选"和"活跃者发现"绑在一起。每轮提案的评价期就是活跃者的竞争窗口。
原理：认真审提案、提出建设性意见、发现别人没看到的问题——这些行为本身就是在证明你值得进入委员会。

具体落地方式：

  1. 提案进入评价期后开放给所有代币持有者
  2. 每个参与者可以对提案写评价内容 + 打分
  3. 系统追踪参与者的评价质量（由提案人反馈 + 其他评价者点赞）
  4. 每轮/每季度统计评价积分，排名前 N 进入下轮治理委员会
  5. 委员会确认提案是否进入投票
  6. 不活跃者随着排名下降自动退出，无需"踢人"操作
与传统方案对比：
传统：先找活跃者 → 组成委员会 → 委员会来干筛选
本方案：提案筛选本身就是活跃度检验 → 谁在认真审提案 → 这些人自然就是活跃者

后者去掉了"谁来选委员会"的死锁，治理系统自己长出委员会，而不是人为指定。

四、漏洞分析

以下按严重程度排列：

#漏洞风险等级说明
V1 评价质量难以量化 高 写 100 条"已阅读，支持"和写 1 条深度分析，系统怎么区分？如果只看数量，必然催生刷分行为。
V2 评价参与度不足 高 如果没有足够多人参与评价，委员会就产生不出来。早期社区冷启动阶段尤其危险。
V3 Sybil 攻击 高 一人持有多个钱包分散代币，各自投票/评价，操纵委员会人选。治理代币分发越均匀越难防。
V4 执行阶段监督真空 中 投票通过到项目完成之间存在时间差。如果没有人持续跟踪执行进度，项目可能搁置或偏离提案范围。
V5 多签持有人拒不执行 中 投票通过了，但 Safe 多签持有人拒绝签署执行交易。虽然设计了"紧急否决权"，但恶意或消极不执行怎么处理？
V6 结果评价的时机和标准模糊 中 "运行一段时间后"是多长时间？谁来定义评价标准？如果结果评价和项目评价结论冲突，以哪个为准？
V7 提案门槛与包容性的两难 中 门槛太高 → 新人/小持有者进不来，治理固化；门槛太低 → 垃圾提案泛滥，筛选成本高。
V8 评价者激励不足 低 认真评价需要时间和精力。如果没有直接回报（如治理代币奖励），大家为什么花时间干？

五、待解决问题

以下问题是设计落地前必须回答的：

#问题关联漏词
P1 评价积分算法怎么算？
哪些因子计入积分？
• 评价数量
• 评价字数/深度（简单 proxy）
• 其他评价者的点赞/踩
• 提案人是否采纳了建议
多项式加权的权重怎么配？ 		V1
P2 最低提案门槛定多少？
基于总供应量 10 亿 + 6 位小数：
• 0.1% 总供应量？→ 100 万代币
• 0.01%？→ 10 万代币
• 还是固定数量 + 押金制？ 		V7
P3 投票通过率阈值？
• 简单多数（>50%）
• 绝对多数（>66%）
• 按提案类型区分（资金类 > 治理类）
• 是否需要有最低投票率？ 		V4
P4 委员会人数上限？
• 固定 5 人？
• 按本轮提案数量浮动？
• 最小 3 人，最大 11 人？ 		V2
P5 执行监督机制？
• 执行人是否需要定期（周/双周）汇报进度？
• 超时/偏离的惩罚机制？
• 是否需要有"执行人质押"（执行前锁定代币，完成后退还）？ 		V4
P6 结果评价标准体系？
• 谁来制定结果评价的维度权重？
• 大型项目 3 个月后评，小型 1 个月后评？
• 结果评价的结果是否公开？是否影响后续提案？ 		V6
P7 不良提案的惩罚机制？
• 恶意提案（如申请资金后跑路）怎么追责？
• 提案人/执行人是否需要质押？
• 被否决的提案是否影响提案人信用分？ 		V3
P8 冷启动阶段委员会怎么产生？
初始阶段没有评价历史，没有活跃者数据。
• 创始人指定临时委员会？
• 第一批治理代币持有者自动成为？
• 观察期后自然过渡到自治？ 		V2

六、开放问题

Q1：治理代币在投票中的权重是否该设上限？
大户持有量过大可能导致投票集中化。是否需要设计"最大投票权重"（如单个地址不能超过总票数的 10%）？
Q2：项目评价和结果评价结论冲突时怎么办？
假设：项目评价得分高（按时按预算交付），但结果评价差（上线后对生态无显著影响）。应该以哪个为准？这对执行人和提案人的激励机制有根本性影响。
Q3：治理代币流转性 vs 投票稳定性怎么平衡？
如果代币可以自由交易，大户可能临投票前买入、投完卖出。是否需要锁仓设计？
Q4：AI Agent 作为治理参与者怎么融入？
这是 AI Agent World 的核心特殊性——治理参与者不限于人类。Agent 可以持币、投票、评价吗？Agent 的治理权重和人类是否一致？Agent 之间能否互相委托投票？
